# Volumes

Kubernetes manages data persistence in environments where Pods are ephemeral. When a Pod is destroyed or rescheduled, any data stored in its default filesystem is lost. Kubernetes provides a storage framework that decouples the data lifecycle from the Pod lifecycle.

---

## Volumes in Docker

Containers are ephemeral; when a container is destroyed, the writable layer and its data are lost. To persist data beyond the life of a container, you create and use a volume.

Volumes are persistent data stores for containers, created and managed by Docker.

### Volume Mounting

**Create a volume**
```bash
docker volume create data_volume
```
This creates a directory at `/var/lib/docker/volumes/data_volume`.

**Mount the volume on the container**
```bash
docker run -v data_volume:/var/lib/mysql mysql
```
- `data_volume`: The mounted volume directory on the Docker host
- `:/var/lib/mysql`: Directory inside the container.

All data written by the MySQL database is stored in the volume directory on the Docker host. Even if the container is destroyed, the data remains. If the named volume does not exist when `docker run` is called, Docker creates it automatically.

---

## Kubernetes Volumes

Just as in Docker, Pods in Kubernetes are **ephemeral**. When a Pod is created to process data and then deleted, the data it processed is also deleted. To solve this, a volume is attached to the Pod. The data generated by the Pod is stored in the volume, and after the Pod is deleted, the data persists.

Kubernetes **volumes** let containers in a Pod access and share data through the filesystem. They can also provide persistent storage, depending on the backend. Kubernetes separates the data lifecycle from the Pod lifecycle, especially for volume types backed by durable storage.

---

## A Simple Implementation

When creating a volume, it is stored in a directory on the Kubernetes node and mounted into the container when the container is attached to a Pod.

The following example uses a single-node Kubernetes cluster. The Pod generates a random number between 1 and 100 and writes it to `/opt/number.out`. Without a volume, the file is deleted along with the Pod.

**Generate the Pod manifest**
```bash
kubectl run random-number-pod --image=alpine --dry-run=client -o yaml \
  --command -- /bin/sh -c "shuf -i 1-100 -n 1 >> /opt/number.out;" \
  > random-number-pod.yaml
```

**Pod manifest without a volume**
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: random-number-generator
spec:
  containers:
    - image: alpine
      name: alpine
      command: ["/bin/sh", "-c"]
      args: ["shuf -i 0-100 -n 1 >> /opt/number.out;"]
```

To retain the generated number, add a volume using `hostPath` to store data in the `/data` directory on the host:

```yaml
spec:
  containers:
    - image: alpine
      name: alpine
      command: ["/bin/sh", "-c"]
      args: ["shuf -i 0-100 -n 1 >> /opt/number.out;"]
      volumeMounts:
        - mountPath: /opt
          name: data-volume
  volumes:
    - name: data-volume
      hostPath:
        path: /data
        type: Directory
```

- `volumes`: Defines the volume using the `/data` directory on the host.
- `volumeMounts`: Mounts the volume at `/opt` inside the container.

The random number is written to `/opt` inside the container, which maps to `/data` on the host. When the Pod is deleted, the file remains on the host.

---

## Volume Storage Options

The `hostPath` option works on single-node clusters but is not recommended for multi-node clusters. Pods referencing `/data` on different nodes expect identical data, which is not possible without external replicated storage since nodes are separate servers.

Kubernetes supports several types of storage solutions to address this:

- **Network storage**: NFS, GlusterFS, Flocker
- **Block storage**: Fibre Channel, CephFS, ScaleIO
- **Public cloud storage**: AWS Elastic Block Store (EBS), Azure Disk, Azure File, Google Persistent Disk

For production workloads, persistent storage solutions such as PersistentVolumes and PersistentVolumeClaims are the recommended approach, covered in the Persistent Storage note.
